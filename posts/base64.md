## Base64란 무엇인가?

웹 개발을 하다 보면 이미지 태그의 `src` 속성에 `https://...` 대신 `data:image/png;base64,iVBORw0KGgo...` 처럼 알 수 없는 긴 문자열이 들어있는 것을 본 적이 있을 것이다. 이것이 바로 **Base64 인코딩**이다.

### 1. 정의 및 동작 원리
Base64 인코딩은 **이진 데이터(Binary Data)를 ASCII 문자열로 변환하는 인코딩 방식**이다.
이름 그대로 **64개의 문자(A-Z, a-z, 0-9, +, /)** 만을 사용하여 데이터를 표현한다.

#### 1-1. 변환 과정 (3 Byte -> 4 Character)
Base64의 핵심 원리는 <b>"3바이트를 4글자로 늘리는 것"</b>이다.

1.  원본 데이터를 **3바이트(24bit)** 씩 끊는다.
2.  이것을 다시 **6비트**씩 4개로 쪼갠다. (6bit * 4 = 24bit)
3.  각 6비트 값을 10진수로 변환한 뒤, Base64 색인표(Index Table)에 매핑되는 문자로 바꾼다.

> **Padding (=)**: 만약 원본 데이터가 3바이트로 딱 떨어지지 않고 남는다면, 빈자리를 등호(`=`) 문자로 채운다. Base64 문자열 끝에 `=`가 보인다면 바로 이것 때문이다.

## 2. 왜 Base64를 사용하는가?

멀쩡한 이진 데이터를 굳이 용량을 늘려가면서 문자열로 바꾸는 이유는 무엇일까?
핵심은 <b>"데이터 전송의 안전성"</b>이다.

### 2-1. 안전한 데이터 전송
통신 과정에서 바이너리 데이터(이미지, 오디오 등)를 그대로 보내면 시스템 간의 차이나 전송 방식에 따라 데이터가 손실되거나 왜곡될 수 있다.
하지만 Base64는 <b>ASCII 문자(텍스트)*</b>로만 이루어져 있기 때문에, 어떤 환경에서도 데이터가 깨지지 않고 안전하게 전송됨을 보장한다.

### 2-2. URL 파라미터 전송 (JWT)
<b>JWT(Json Web Token)</b>가 대표적인 예시다. JWT는 헤더나 URL 파라미터로 전송되는데, 여기에는 특수문자(`+`, `/`, `=`)가 포함되면 안 된다.
그래서 JWT는 표준 Base64에서 `+`와 `/`를 URL에서 안전한 문자(`-`, `_`)로 대체하고 패딩(`=`)을 제거한 **URL-Safe Base64** 방식을 사용한다.

## 3. 웹 성능 최적화: 이미지를 Base64로?

프론트엔드 개발에서 Base64는 성능 최적화 기법으로도 사용된다. 이를 **Data URI Scheme**라고 한다.

### 3-1. 장점: HTTP 요청 감소
이미지를 별도 파일로 로딩(`src="image.png"`)하면 브라우저는 서버에 HTTP 요청을 보내야 한다.
하지만 이미지를 Base64 문자열로 변환하여 HTML/CSS 파일 내부에 직접 박아넣으면(Inlining), **추가적인 HTTP 요청 없이 이미지를 렌더링**할 수 있다.

*   **활용 예시**: 아주 작은 아이콘, 로고, 배경 패턴 등.
*   **Webpack/Vite**: 번들러 설정에서 일정 크기(예: 8KB) 이하의 이미지는 자동으로 Base64로 변환하여 번들링하도록 설정하는 경우가 많다.

### 3-2. 치명적인 단점: 용량 증가 및 캐싱 불가
하지만 무조건 좋은 것은 아니다. 명확한 단점이 존재한다.

1.  **용량 증가**: 앞서 말했듯 3바이트가 4바이트가 되므로, 데이터 크기가 **약 33% 증가**한다. 큰 이미지를 변환하면 HTML/CSS 파일 자체가 너무 커져서 오히려 초기 로딩이 느려질 수 있다.
2.  **캐싱 불가**: 일반 이미지 파일은 브라우저가 캐싱하여 재사용하지만, HTML 안에 포함된 Base64 데이터는 **문서와 한 몸이므로 별도로 캐싱할 수 없다.** 같은 이미지가 여러 페이지에 쓰인다면 매번 다운로드해야 하므로 비효율적이다.

### 4. 결론: 언제 써야 할까?

| 구분 | Base64 (Data URI) 사용 권장 | 이미지 파일 URL 사용 권장 |
| :--- | :--- | :--- |
| **파일 크기** | 매우 작음 (수 KB 이하) | 큼 |
| **사용 빈도** | 일회성 혹은 특정 페이지 전용 | 여러 페이지에서 공통 사용 |
| **목적** | HTTP 요청 횟수 줄이기 (네트워크 오버헤드 감소) | 브라우저 캐싱 활용 |

Base64는 **"네트워크 요청 횟수를 줄이는 대신 데이터 전송량을 늘리는 트레이드오프(Trade-off)"** 기술이다. 상황에 맞춰 적절히 사용하는 지혜가 필요하다.
